# Завдання 1. Створення Dockerfile

Для контейнеризації курсового проєкту Spring Boot було створено файл Dockerfile.

Dockerfile:

```dockerfile
FROM maven:3.9.9-eclipse-temurin-21 AS build

WORKDIR /app

COPY pom.xml .

COPY src ./src

RUN mvn clean package -DskipTests

FROM eclipse-temurin:21-jre

WORKDIR /app

COPY --from=build /app/target/*.jar app.jar

EXPOSE 8080

ENTRYPOINT ["java","-jar","app.jar"]
```

Для збірки було використано образ Maven з JDK 21, а для запуску — легкий образ Eclipse Temurin JRE 21. Такий підхід дозволяє зменшити розмір фінального Docker-образу.

# Відповіді на питання

## 1. Навіщо потрібен multi-stage build?

Multi-stage build дозволяє розділити процес збірки та запуску застосунку на декілька етапів. У фінальний образ потрапляє лише готовий jar-файл, тому образ займає менше місця.

## 2. Для чого потрібен .dockerignore?

Файл .dockerignore виключає непотрібні файли та каталоги з контексту збірки Docker. Без нього до образу могли б потрапити службові файли IntelliJ IDEA та артефакти збірки.

## 3. Чим відрізняються CMD і ENTRYPOINT?

CMD задає команду за замовчуванням і може бути змінена під час запуску контейнера. ENTRYPOINT визначає основну команду запуску контейнера та використовується для запуску застосунку.

## 4. Чому застосунок має слухати 0.0.0.0?

Адреса 0.0.0.0 дозволяє приймати підключення з будь-якого мережевого інтерфейсу контейнера. Якщо використати localhost або 127.0.0.1, застосунок буде доступний лише всередині контейнера.

## 5. Чому Docker Hub зручніший?

Docker Hub дозволяє швидко отримати готовий образ без встановлення залежностей та налаштування середовища. Крім того, усі користувачі запускають однакову версію застосунку.
